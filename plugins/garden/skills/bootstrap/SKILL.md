# Bootstrap a GARDEN project

Read `references/principles.md` relative to the plugin root before applying the GARDEN rules. Read the sibling `references/glossary.md` before applying terms such as vertical slice, managed duplication, or significant directory.

Treat every mechanizable MUST rule as a gate requirement. Hooks and deterministic gates enforce a rule far more reliably than an instruction file alone. Keep prose for intent; encode enforceable rules in types, lint, tests, hooks, or CI. [CLAIM-N001]

Complete the steps in order. Do not begin the next step while its acceptance signal fails; correct the current artifact first.

1. Choose a slice map before writing code.

   Action: List each required capability as a flat plan of one directory per capability. Make every directory an atomic vertical slice owning its entry point, logic, data access, and tests.

   Acceptance signal: Every planned directory maps to exactly one capability, and no capability spans more than one directory.

2. Set the naming registry.

   Action: Create `naming-registry.txt` at the project root before the first slice. Add exactly one `concept: canonical-name` entry per line for every domain concept introduced.

   Acceptance signal: Every planned identifier, file name, and directory name traces to one registry entry; no concept has a synonym.

3. Write the root context file.

   Action: Hand-write the root `CONTEXT.md`, at most 200 lines. State the naming-registry location, slice layout, contract locations, and deterministic-gate locations. Permit `AGENTS.md` or `CLAUDE.md` only as a symlink or copy of `CONTEXT.md`.

   Acceptance signal: `CONTEXT.md` is within 200 lines, contains no autogenerated bulk dump, and lets a new human or agent reach every project document within one hop.

4. Scaffold one directory per slice.

   Action: Create every capability directory directly at the project root and add a `README.md` whose first paragraph states the slice's purpose, contract, and related slices. Do not add a `slices/` wrapper. If the project requires another slice root, record that choice in `CONTEXT.md`. Add a `README.md` to every significant directory as defined in the plugin's `references/glossary.md`.

   Acceptance signal: Every planned slice has its root-level directory and README, and each README states its purpose in the first paragraph.

5. Define contracts before code.

   Action: Before writing an entry point or logic, add a `CONTRACT.md` beside each slice's future code. Make its first line exactly `Version: MAJOR.MINOR.PATCH`, then state the interface, observable behavior, examples, inputs, outputs, and errors needed to regenerate the slice.

   Acceptance signal: A reviewer unfamiliar with the implementation can predict each slice's observable behavior from its contract alone.

6. Wire deterministic gates from day one.

   Action: Configure type checking, linting, unit tests, contract tests, and clone detection as fail-fast CI gates before the codebase grows beyond the first slice. Treat lint configuration as executable architecture rather than a convention.

   Acceptance signal: A stated invariant fails CI without relying on a human or agent to notice it manually.

7. Set the clone-detection budget.

   Action: Configure clone detection to flag blocks duplicated three or more times. Keep the default threshold at 3; record any override in `CONTEXT.md`. Do not extract an abstraction before three concrete usages justify it.

   Acceptance signal: Clone detection runs in CI, produces a visible report, and the project has no abstraction extracted solely for fewer than three usages.

## Generate stack-specific gates

Detect the target project's language, package manager, build files, test runner, and existing CI configuration. Generate only stack-compatible artifacts; do not assume a language or toolchain.

Generate, at minimum:

- A linter configuration that acts as an executable architecture specification. Enforce naming and layering where supported, including this import boundary: a slice may import another slice's contract, never its internals.
- A fail-fast CI pipeline in this order: type-check, lint, unit-tests, contract-tests, clone-detection. Keep type-check through unit-tests strictly sequential. Permit clone-detection to run in parallel with contract-tests, but only after unit-tests pass.
- A clone-detection job with threshold 3 unless `CONTEXT.md` records an explicit override.

Read the plugin's `references/principles.md` again when selecting each enforceable rule. Convert every mechanizable MUST rule into the strongest stack-supported type, lint, test, hook, or CI gate; document an invariant that cannot be made deterministic as a risk rather than leaving it as prose.

Mark any agent-drafted "why" in `CONTEXT.md` or a contract as a draft until a human authors or approves it. Do not represent agent-generated intent, trade-offs, or decision records as human-approved.
