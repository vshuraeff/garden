# GARDEN project rules

## Discoverability and slices

- Maintain `naming-registry.txt` with one `concept: canonical-name` entry per line.
- Use canonical names in code, tests, documentation, errors, files, and directories. Keep domain call sites statically searchable.
- Put capability directories at the project root, one directory per vertical slice. Use another slice root only when `CONTEXT.md` records that explicit choice.
- Keep each slice focused on one capability and colocate its tests. Do not introduce an abstraction before three concrete usages exist.
- Keep dependencies, boundary shapes, ordering, meaningful literals, and errors explicit. Do not rely on hidden globals, temporal coupling, or magic values.

## Contracts and knowledge

- Put a `CONTRACT.md` beside each component. Its first line must be exactly `Version: MAJOR.MINOR.PATCH`.
- Keep a hand-written `CONTEXT.md` at the project root, at most 200 lines. Do not add autogenerated bulk dumps.
- Add a README to every significant directory: one containing a `CONTRACT.md`, with 2 or more subdirectories, or with more than 5 source files.
- Record any skipped or deferred gate or rule in `retrofit-log.md` with its reason.

## Deterministic verification

- Run CI gates in this order: type-check, lint, unit-tests, contract-tests, clone-detection. Clone-detection may run in parallel with contract-tests, but only after unit-tests pass.
- Configure clone-detection to flag duplication of 3 or more occurrences. Record any threshold override in `CONTEXT.md`.
- Encode invariants in types, lint, tests, or CI gates. An agent must not self-certify a change; deterministic gates decide.
